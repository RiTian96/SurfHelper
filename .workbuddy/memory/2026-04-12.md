# 2026-04-12 工作日志

## fongmi-apk-updater.bat 优化（多轮迭代）

### 第一轮：基础 bug 修复
- 修复 `::START_DOWNLOAD` / `::DOWNLOAD_ABORT` 标签语法，改为单冒号有效标签
- 修复 `cd /d "%~dp0"` 位置问题，移到 `:START_DOWNLOAD` 标签后
- 修复 BASE_URL 缺少 `https://` 前缀
- 下载 curl 加 `--connect-timeout 10 --max-time 300`

### 第二轮：测速逻辑重写
- 原来"找到第一个通的就用"导致前几个快节点被 HEAD 请求误判跳过
- 重写为全部测完选最快：用 GET 请求 + `speed_download` 测真实下载速率
- 测速用实际 APK 文件（4 秒下载实测吞吐量），而非 favicon 小文件
- 编码修复：文件保存为 UTF-8 无 BOM + `chcp 65001`（BOM 会导致 `锘` 乱码）

### 第三轮：并行测速 + 两阶段筛选
- 串行测速问题：第一个节点占满带宽，后面的节点测出来都慢
- 改为 `start /b` 并行下载同一 APK，6 秒后比谁下载的文件最大
- 编码教训：`.bat` 文件必须 UTF-8 无 BOM + `chcp 65001`；子程序内注释用 `REM` 不能用 `::`

### 第四轮：大量节点支持（57 个）+ 竞速模式最终方案
- 节点列表已去重（mirrors.chenby.cn、github.geekery.cn 等重复项已合并）
- 默认备用线路改为 gh.xxooo.cf
- 最终方案：57 节点并行竞速（5MB 阈值）→ 胜出节点 3 文件并行下载
- bat 编码：UTF-8 无 BOM + `chcp 65001`；子程序内注释用 `REM`；for 循环内禁用 `goto`
- bat 关键坑：`::` 标签是注释不是 goto 目标；for 内 goto 会跳出整个循环；taskkill 会杀掉所有 curl 进程

### 第五轮：精简为 4 个优质节点 + 逐文件容错下载
- 用 test-nodes.ps1 串行测试 57 个节点（3 轮取平均），筛选出 4 个优质节点：
  gh.xxooo.cf / gh.llkk.cc / ghfile.geekertao.top / github.tbedu.top（均 ~5.4 MB/s）
- 去掉竞速阶段，直接逐文件下载，按节点优先级尝试
- 卡住检测：每 4 秒检查文件大小，8 秒不增长则判定卡住，自动换下一个节点
- 卡住时 taskkill curl → 换下一个节点重试
- 测试脚本：test-nodes.ps1 (PowerShell 串行测速) + test-nodes.bat (启动器)
